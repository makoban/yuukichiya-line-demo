## 1. 根本原因（最重要・恒久知識）

### 症状
リッチメニューの各ボタン（会員情報/ポイント/採寸記録/クーポン）を押すと、**一瞬正しい画面が出てから、全部「採寸予約」に化ける**。

### 真因
LIFF の **Endpoint URL** が以下になっていた（汚染）:
```
https://makoban.github.io/yuukichiya-line-demo/?screen=reservation   ← 誤
```
正しくは**クエリ無しの中立URL**:
```
https://makoban.github.io/yuukichiya-line-demo/                       ← 正
```
- 場所: LINE Developers → **LINEログインチャネル `2010371637`**（Messaging APIチャネル `2010371574` ではない）→ LIFFタブ → `2010371637-PcIXzbgC` → エンドポイントURL。
- LIFF IDの「-」より前の数字 `2010371637` が、そのLIFFを持つチャネルID。

### なぜ気づきにくかったか
- 普通のブラウザでは `liff.isInClient()` が false で `liff.init()` が走らないため、アプリは `liff.state` を正しく読み、**会員照会が正常表示される（＝Safari等では再現しない）**。
- LINEアプリ内だけ `liff.init()` が `liff.state` を消化し、土台に残った `?screen=reservation` が勝つ → 採寸予約に化ける。
- アプリ側は起動時に画面を1回だけ決定し、既定は `member`（[app.js](app.js) の `openInitialScreenFromUrl`）。コードからは絶対に "reservation" を生み出せない → 汚染はエンドポイント側、と特定できた。

### 検証方法（ログイン不要・ブラウザだけ）
普通のブラウザで次を開き、**リダイレクト先の土台に `?screen=...` が無い**（`?liff.state=...` だけ）ことを確認:
```
https://liff.line.me/2010371637-PcIXzbgC/?screen=member
→ 正: https://makoban.github.io/yuukichiya-line-demo/?liff.state=%3Fscreen%3Dmember
→ 誤（汚染時）: https://makoban.github.io/yuukichiya-line-demo/?screen=reservation&liff.state=...
```

## 4. 現在のリッチメニュー アクションURL（A–F）

LINE Official Account Manager（`https://manager.line.biz/account/@528mzivr/richmenu`）の各タップ領域に設定する想定値:

| 枠 | 表示名 | URL |
| --- | --- | --- |
| A | 会員情報 | `https://liff.line.me/2010371637-PcIXzbgC/?screen=member` |
| B | ポイント | `https://liff.line.me/2010371637-PcIXzbgC/?screen=points` |
| C | 採寸予約 | `https://liff.line.me/2010371637-PcIXzbgC/?screen=reservation` |

## 6. デプロイ方法

- `main` に push すると GitHub Pages（`https://makoban.github.io/yuukichiya-line-demo/`）へ反映される（このリポジトリの通常運用）。
- 静的ファイルは `index.html` の `?v=` でキャッシュバストする。**ファイルを変更したら必ず該当の `?v=` を上げる**（styles.css / app.js）。これを怠るとLINE内ブラウザが旧版をキャッシュし「直したのに変わらない」が再発する。
- 反映後はスマホの**LINEを完全終了→再起動**してキャッシュを更新。
